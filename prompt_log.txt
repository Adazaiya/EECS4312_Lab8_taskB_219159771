Prompt 1:
I am building an Event Registration with Waitlist system in Python. The system
manages registration for a single fixed-capacity event. It must register users
until capacity is reached, place additional users on a FIFO waitlist, and
automatically promote the earliest waitlisted user when a registered user cancels.

The system has two personas:
- Jane: a busy professional who needs automatic conflict resolution, predictable
  ordering, and minimal interruptions. She is frustrated by duplicate notifications
  and inconsistent behavior.
- Mo: a user with accessibility needs who requires explicit, descriptive feedback
  for every system action, deterministic behavior under all conditions, and clear
  explanations when the system promotes or rejects a user.

Constraints derived from the personas:
- C1: registered count must never exceed capacity
- C2: user_id must be a non-empty, non-whitespace string
- C3: a user may not appear more than once in the system
- C4: a user may not exist in both registered and waitlist states simultaneously
- C5: waitlist must preserve strict FIFO ordering
- C6: promotion must be fully deterministic
- C7: every error must raise a named typed exception with a descriptive message
- C8: system state must remain consistent after every operation
- C9: snapshot() must include a last_action field describing the most recent operation
- C10: each operation must trigger at most one state-change outcome

Acceptance criteria:
- AC1: register() returns UserStatus("registered") when slots are available
- AC2: register() returns UserStatus("waitlisted", N) when event is full
- AC3: cancel() promotes earliest waitlisted user and last_action reflects this
- AC4: cancel() of waitlisted user removes them without triggering promotion
- AC5: duplicate register raises DuplicateRequest, state unchanged
- AC6: capacity=0 places all users on waitlist, registered_count stays 0
- AC7: cancel() of unknown user raises NotFound with descriptive message
- AC8: invalid user_id raises ValueError before any state is modified
- AC9: snapshot() last_action describes the most recent operation
- AC10: identical operation sequences produce identical outcomes

Please review this specification and identify any missing constraints, ambiguous
requirements, or unhandled edge cases. Do not generate code yet.

Response summary:
The LLM identified the following issues:
1. The last_action field was not present in the Lab 8 snapshot() — it needed to
   be added as a new constraint (C9) to satisfy Mo's transparency need.
2. The DuplicateRequest and NotFound exception messages were not descriptive
   enough in Lab 8 — they needed to include the user_id and reason.
3. The case where cancel() finds no waitlisted user to promote was not explicitly
   handled in last_action — the LLM flagged this as an unhandled edge case.
4. The LLM confirmed all other constraints were complete and unambiguous.

Decision: Accept — added C9, updated FR3 and FR4, added EC2 for duplicate
cancellation, and documented all changes in Revision Notes.
