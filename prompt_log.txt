Prompt #1
================================================================================

PURPOSE: Provide full requirements to the LLM and ask for ambiguity check
         before any code generation begins.

PROMPT:
  I am building a Python module for an Event Registration with Waitlist system.
  Please review my requirements below. Before generating any code, identify
  any ambiguities or missing clarifications.

  --- PROBLEM SUMMARY ---
  A Python module that manages registration for a single event with a fixed
  capacity. Users can register, be placed on a waitlist, cancel, and query
  their status.

  --- FUNCTIONAL REQUIREMENTS ---
  FR-01: The system shall accept a fixed non-negative integer capacity at
         initialization. If capacity is negative, raise a ValueError.
  FR-02: The system shall register users in arrival order until the registered
         list reaches the capacity limit.
  FR-03: When the registered list is at capacity, additional users shall be
         placed onto a FIFO waitlist.
  FR-04: When a registered user cancels, the system shall automatically promote
         the user at the front of the waitlist (if any) to registered status.
  FR-05: When a waitlisted user cancels, the system shall remove them from the
         waitlist without triggering promotion.
  FR-06: The system shall raise DuplicateRequest if a user attempts to register
         while already registered or waitlisted.
  FR-07: Any user may query their current status: registered, waitlisted
         (with 1-based position), or none.
  FR-08: If capacity is zero, all users who attempt to register shall be placed
         directly on the waitlist.
  FR-09: Cancellation of a user not in the system shall raise NotFound.

  --- CONSTRAINTS ---
  C-01: The number of registered users shall never exceed capacity.
  C-02: user_id must be a non-empty, non-whitespace string.
  C-03: A user may not appear more than once in the system.
  C-04: A user may not simultaneously exist in both registered and waitlisted states.
  C-05: Waitlist ordering shall preserve strict FIFO behavior.
  C-06: Promotion is deterministic under identical operation sequences.
  C-07: capacity is fixed at initialization and cannot change.
  C-08: System state must remain consistent after every operation.

  --- ACCEPTANCE CRITERIA ---
  AC-01: Given capacity > 0 and slots available, when a user registers,
         they appear in registered with state="registered".
  AC-02: Given event is full, when a user registers, they are placed on
         the waitlist with correct 1-based position.
  AC-03: Given a registered user cancels, the earliest waitlisted user
         is promoted to registered.
  AC-04: Given a waitlisted user cancels, no promotion occurs.
  AC-05: Given a duplicate register attempt, DuplicateRequest is raised.
  AC-06: Given capacity=0, all registrations go to waitlist.
  AC-07: Given a cancelled user re-registers, they enter at the rear of
         the waitlist if full, or registered if slots are available.
  AC-08: Given an invalid user_id, ValueError is raised.
  AC-09: Given a cancel for unknown user, NotFound is raised.

  Please list any ambiguities or missing information before writing any code.

--------------------------------------------------------------------------------
RESPONSE SUMMARY:
  The model identified the following ambiguities:

  1. THREAD SAFETY: Does the system need to handle concurrent access?
     -> RESOLVED: System is single-threaded. Rapid operations means sequential
        calls in the same thread.

  2. CASE SENSITIVITY: Is "Alice" the same user as "alice"?
     -> RESOLVED: user_id comparison is case-sensitive.

  3. MAX WAITLIST SIZE: Is there a limit to how many users can be waitlisted?
     -> RESOLVED: No maximum waitlist size. Users may queue indefinitely.

  4. RETURN TYPE OF cancel(): Should it return anything?
     -> RESOLVED: cancel() returns None. All outcomes communicated via exceptions.

  5. RE-REGISTRATION: Can a user re-register after cancelling?
     -> RESOLVED: Yes, a cancelled user has no record and may re-register
        as a new arrival.

DECISION: ACCEPT — all ambiguities resolved. Proceeding to code generation.
